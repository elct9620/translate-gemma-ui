# TranslateGemma UI

## Purpose

提供一個本地運行的翻譯工具，使用開放權重模型 TranslateGemma 4B，讓使用者在不依賴雲端服務的情況下完成多語言翻譯與字幕在地化工作。

## Users

| 使用者 | 需求 |
|--------|------|
| 影音工作者 | 翻譯 SRT 字幕檔，維持詞彙一致性 |
| 字幕翻譯者 | 批次翻譯字幕，搭配詞彙表控制譯詞 |
| 一般使用者 | 簡單的文字段落翻譯 |

使用者不一定具備工程背景，安裝與操作流程需簡單直覺。

## Impacts

- 翻譯過程完全在本地執行，保護內容隱私
- 透過詞彙表確保翻譯用詞一致
- 降低對付費翻譯 API 的依賴

## Success Criteria

- 使用者可在 macOS 或 Windows 上完成安裝並啟動應用程式
- 使用者可透過 GUI 完成一份 SRT 字幕檔的翻譯並下載結果
- 在 4GB VRAM 的 GPU 上，系統可載入模型並完成翻譯
- 詞彙表中指定的譯詞出現在翻譯結果中

## IS / IS NOT

| IS | IS NOT |
|----|--------|
| 本地桌面應用程式 | 雲端翻譯服務 |
| 批次字幕翻譯工具 | 即時語音翻譯 |
| 使用單一模型（TranslateGemma 4B） | 多模型切換平台 |
| 支援 SRT 格式 | 通用字幕格式轉換器 |

## Features

1. **文字翻譯**：使用者輸入文字，選擇來源與目標語言，系統輸出翻譯結果
2. **SRT 字幕翻譯**：使用者上傳 SRT 檔案，選擇翻譯模式（整檔或批次），系統翻譯並保留時間軸，輸出翻譯後的 SRT 檔案
3. **詞彙表**：使用者提供詞彙對照表，系統在翻譯時優先採用指定譯詞
4. **GUI 介面**：透過瀏覽器操作的圖形介面

## User Journeys

### 文字翻譯

| Context | Action | Outcome |
|---------|--------|---------|
| 使用者開啟應用程式 | 輸入文字、選擇來源語言與目標語言，按下翻譯 | 系統顯示翻譯結果 |

### SRT 字幕翻譯

| Context | Action | Outcome |
|---------|--------|---------|
| 使用者有一份英文 SRT 字幕檔 | 上傳 SRT 檔案、選擇目標語言、選擇翻譯模式、選擇性上傳詞彙表與替換方式，按下翻譯 | 系統依所選模式翻譯，產生翻譯後的 SRT 檔案供下載，時間軸不變 |

### 詞彙表管理

| Context | Action | Outcome |
|---------|--------|---------|
| 使用者希望特定詞彙有固定翻譯 | 上傳 CSV 格式詞彙表（來源詞, 目標詞），選擇替換方式（翻譯前或翻譯後） | 系統依所選方式在翻譯前或翻譯後替換指定詞彙 |

## Behaviors

### 模型載入

- 系統啟動時嘗試載入 `google/translategemma-4b-it` 模型
- 載入失敗時，系統退回開發模式（FakeTranslator），介面顯示模型載入失敗訊息
- 介面提供 HF Token 輸入欄位與「載入模型」按鈕，使用者可輸入 Hugging Face 存取權杖後重新載入模型
- 模型在首次使用時下載並快取至本地

### 語言選擇

- 系統支援的語言為固定清單：英語（en）、繁體中文（zh-TW）、日語（ja）
- 來源語言與目標語言不得相同

### 文字翻譯

- 使用者輸入文字，選擇來源與目標語言，按下翻譯後系統輸出翻譯結果
- 模型上下文長度從模型組態（tokenizer config）動態讀取
- 無硬性輸入長度限制，當輸入未超過模型上下文長度時，系統一次完成翻譯
- 當輸入超過模型上下文長度時，系統自動以 Sliding Window 方式分段翻譯：
  - 分段點：以句子邊界為分段點，每段不超過模型上下文長度。句子邊界定義為以下標點符號：句號（.、。）、問號（?、？）、驚嘆號（!、！）
  - 重疊策略：相鄰段落重疊 2 句，確保翻譯連貫性
  - 合併策略：重疊區域以後段翻譯結果為準（後段擁有更完整的前文上下文）
  - 進度指示：分段翻譯時，系統顯示進度（已完成段數 / 總段數）
- 翻譯結果以串流方式逐步顯示，使用者可在翻譯完成後複製完整結果

### SRT 翻譯

- 系統解析 SRT 檔案，保留原始時間軸與序號
- 使用者可選擇翻譯模式：
  - **整檔模式**：將 SRT 全部內容（含時間軸與格式）整體送入模型，由模型保留格式並翻譯文字部分，一次完成
  - **批次模式**：每次合併 N 句字幕文字一起送入模型翻譯，使用者可設定 N 值（預設 1，即逐句翻譯）
- 整檔模式適合字幕量較少或希望模型自行維持上下文連貫的場景；批次模式適合字幕量較大、需要逐步確認進度或避免超過模型輸入長度的場景
- 翻譯過程中，系統顯示進度指示（整檔模式顯示翻譯中/完成；批次模式顯示已完成批次數 / 總批次數）
- 翻譯完成後，系統產生新的 SRT 檔案，格式與原檔一致

### 詞彙表

- 詞彙表格式為 CSV，規格如下：

| 項目 | 定義 |
|------|------|
| 編碼 | UTF-8 |
| 分隔符 | 逗號 |
| 標頭列 | 無，第一列即為資料 |
| 欄位 | 第一欄：來源語言詞彙，第二欄：目標語言詞彙 |

- TranslateGemma 不支援在提示中注入詞彙指令（見「模型限制」），因此詞彙表以應用層文字替換方式實現
- 使用者可選擇替換時機：
  - **翻譯前替換（Pre-processing）**：翻譯前將原文中匹配的來源詞替換為目標詞，模型直接翻譯已替換的文字
  - **翻譯後替換（Post-processing）**：翻譯後掃描結果中殘留的來源詞，替換為目標詞
- 拉丁字母詞彙以完整詞彙比對（word boundary），非拉丁字母詞彙以子字串比對，皆不區分大小寫（case-insensitive）
- 詞彙表為選填，未提供時系統正常翻譯

### 模型限制

- TranslateGemma 為專門的結構化翻譯模型，僅接受 `type`、`source_lang_code`、`target_lang_code`、`text` 四個欄位
- 模型不支援自訂系統提示、上下文注入、或詞彙表指令等 prompt engineering 技巧
- 系統使用官方 `apply_chat_template` API 與模型互動，不使用自訂提示格式

### GUI 介面

- 系統提供基於瀏覽器的圖形介面
- 預設使用 Gradio 框架
- 系統在介面上顯示目前使用的運算裝置資訊：
  - 裝置類型（如 NVIDIA RTX 4090、Apple M2 GPU、CPU）
  - 可用 VRAM（GPU）或可用記憶體（CPU）
- 使用 CPU 運算時，系統額外顯示提示：翻譯速度可能較慢

## Error Scenarios

| 情境 | 系統行為 |
|------|---------|
| 模型載入失敗（無 Token 或 Token 無效） | 退回開發模式，介面顯示失敗訊息並引導使用者透過 UI 輸入 HF Token 後重新載入 |
| 上傳的檔案非合法 SRT 格式（見 Terminology 中 SRT 格式定義） | 顯示格式錯誤訊息，拒絕處理 |
| 來源語言與目標語言相同 | 禁止送出翻譯請求 |
| 文字翻譯輸入內容為空白 | 禁止送出翻譯請求 |
| 詞彙表 CSV 格式錯誤 | 顯示格式錯誤訊息，提示正確格式 |
| 模型下載失敗（網路問題） | 顯示下載失敗訊息，提示檢查網路連線與 Token |
| 翻譯過程中發生錯誤（文字翻譯） | 顯示錯誤訊息，錯誤訊息旁顯示「重試」按鈕，點擊後從失敗處繼續翻譯 |
| 翻譯過程中發生錯誤（SRT 批次模式） | 該批次保留原文，繼續翻譯後續批次；使用者可下載部分翻譯結果（已翻譯句為譯文，未翻譯句保留原文） |
| 翻譯過程中發生錯誤（SRT 整檔模式） | 顯示錯誤訊息，使用者可重試或改用批次模式 |
| 整檔模式輸入超過模型上下文長度 | 顯示提示訊息，建議使用者改用批次模式 |
| GPU 偵測失敗或不支援 | 自動退回 CPU 模式，介面顯示目前使用 CPU 運算 |
| Sliding Window 分段翻譯時某一段失敗 | 該段保留原文，繼續翻譯後續段落，最終結果中失敗段落維持原文（不額外標記，與正常輸出格式一致） |
| 模型推論時記憶體不足（OOM） | 顯示記憶體不足錯誤訊息，建議使用者關閉其他應用程式或改用 CPU 模式 |
| SRT 檔案包含空白字幕行 | 保留空白行的時間軸與序號，翻譯結果中該行維持空白 |
| 詞彙表中的詞彙語言與翻譯方向不匹配 | 系統忽略不匹配的詞彙對，正常完成翻譯 |
| 模型尚未載入完成時使用者嘗試翻譯 | 禁止送出翻譯請求，顯示模型載入中提示 |
| 翻譯進行中使用者離開或重新整理頁面 | 翻譯中斷，已完成部分不保留，使用者可重新上傳並翻譯 |
| 使用者同時發起多個翻譯請求 | 系統同時僅處理一個翻譯請求，後續請求排隊等待，介面顯示目前排隊狀態（如「前方 N 個請求等待中」） |

## Terminology

| 術語 | 定義 |
|------|------|
| TranslateGemma | Google 開放權重翻譯模型（4B 參數版本） |
| SRT | SubRip Subtitle 格式，純文字字幕檔案格式。合法 SRT 由多個字幕區塊組成，每個區塊包含：序號（正整數）、時間軸（`HH:MM:SS,mmm --> HH:MM:SS,mmm`）、一或多行字幕文字，區塊間以空行分隔 |
| HF Token | Hugging Face 平台的 API 存取權杖，使用者透過介面輸入 |
| 詞彙表 | CSV 格式的來源詞與目標詞對照表 |
| 運算裝置 | 執行模型推論的硬體，可為 GPU（含 VRAM）或 CPU（使用系統記憶體） |
| Sliding Window | 當文字超過模型上下文長度時，將文字分段並保留段落間重疊部分以維持上下文連續性的翻譯策略 |
| Pre-processing | 翻譯前替換——在送入模型前，將原文中匹配的詞彙替換為目標譯詞 |
| Post-processing | 翻譯後替換——在模型輸出後，將結果中殘留的來源詞替換為目標譯詞 |
| 整檔模式 | SRT 翻譯模式之一，將整份 SRT 內容（含時間軸）一次送入模型翻譯 |
| 批次模式 | SRT 翻譯模式之一，每次合併 N 句字幕文字送入模型翻譯，N 由使用者設定 |

## Environment

| 項目 | 需求 |
|------|------|
| Python | >= 3.12 |
| 作業系統 | macOS、Windows |
| 套件管理 | uv（主要）、pip + requirements.txt（備選） |
| 模型 | google/translategemma-4b-it（需 HF Token） |
| 最低 GPU | NVIDIA GTX 3050（4GB VRAM） |
| 無 GPU 時 | 系統退回 CPU 模式，功能不受限，速度較慢 |

### 硬體適應

- 系統以 4GB VRAM 的低階 GPU（如 GTX 3050）作為基準硬體目標
- 在 VRAM 不足以載入完整模型時，系統自動以降低精度方式載入模型（如量化），使模型可在有限 VRAM 中運行
- 無可用 GPU 時，系統自動退回 CPU 模式，所有功能維持可用
